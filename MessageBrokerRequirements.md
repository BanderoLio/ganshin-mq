# AMQP БРОКЕР СООБЩЕНИЙ

## ЦЕЛЬ ПРОЕКТА
Разработать собственный брокер сообщений с базовой функциональностью pub/sub, очередями сообщений.

## ОПИСАНИЕ ЗАДАЧИ
Необходимо реализовать брокер сообщений, который будет обеспечивать асинхронную коммуникацию между тремя тестовыми микросервисами (один publisher, один subsriber).

должно быть полностью совместимо с amqplib на nodejs

## ОСНОВНЫЕ ТРЕБОВАНИЯ

### 1. БАЗОВАЯ ФУНКЦИОНАЛЬНОСТЬ (Обязательно)
- Реализация паттерна Publisher/Subscriber
- Поддержка очередей сообщений (FIFO, LIFO ...)
- Возможность публикации сообщений в топики/очереди
- Возможность подписки на топики/очереди
- Обработка множественных подписчиков на один топик/очередь

### 2. ПЕРСИСТЕНТНОСТЬ (Обязательно)
- Сохранение сообщений на диск (файловая система или БД)
- Восстановление состояния после перезапуска
- Гарантия доставки сообщений (at-least-once или exactly-once)

### 3. КЛИЕНТСКАЯ БИБЛИОТЕКА (Обязательно)
- SDK для любого ЯП (amqplib уже есть)

### 4. ДОПОЛНИТЕЛЬНЫЕ ФУНКЦИИ
- Приоритеты сообщений
- TTL (Time To Live) для сообщений
- Dead Letter Queue (DLQ)
- Метрики и мониторинг (Prometheus endpoint)
- Аутентификация и авторизация
- Кластеризация/репликация
- Web UI для мониторинга
- любые другие одобренные мной

## ТЕХНИЧЕСКИЕ ТРЕБОВАНИЯ

- **Язык программирования**: C++
- **Хранение данных**: файловая система, SQL, или любая NoSQL БД
- **Протокол взаимодействия**: AMQP
- **Документация**: Полная архитектурная схема, полная документация всех модулей системы
- **Деплой**: Docker контейнер с docker-compose (сборки под linux/freebsd/unix ОС)



## ВОЗМОЖНЫЕ ФИЧИ

- Реализация транзакций (publish в несколько топиков атомарно)
- Оптимизация производительности (любой интересный способ, не просто в рамках кода)
- Поддержка схем сообщений (Protobuf...)
